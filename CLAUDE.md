# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Project Is

**Design Studio for FastLED** is a browser-based node-graph editor for designing LED lighting effects. Users wire together nodes on a canvas, see a real-time LED matrix preview, then generate FastLED/Arduino C++ firmware for upload to microcontrollers like the ESP32-S3.

Core user flow: drag nodes from sidebar в†’ wire ports together в†’ preview updates live в†’ generate C++ в†’ upload via the local build helper (`arduino-cli`/`fbuild`).

## Public beta вЂ” protect saved projects

The project is public and beta users may have saved projects, shared graphs, and
pattern libraries. Changes that rename, remove, or reshape node types,
properties, port ids, or persisted workspace data must either preserve existing
saves or include a deliberate migration and release note. Do not add speculative
migration layers, but do not assume external data can be discarded. Update tests
for both the new schema and any supported compatibility path.

## Git workflow (public repository)

This is a public repository. Keep `main` releasable and make every change on a
short-lived branch reviewed through a pull request. **These instructions
override the global cortex / strict-git rules for this repository.**

- **Use plain `git`** (`git add`, `git commit`, `git push`, `git pull`). Do **not**
  use `cortex git` here.
- **Never commit or push directly to `main`.** Start from an up-to-date `main`,
  then create a descriptive branch before editing. Codex branches use the
  `codex/` prefix; other contributors may use `fix/`, `feature/`, `docs/`, or
  another clear prefix.
- Keep branches focused on one change. Push the branch with upstream tracking
  and open a pull request into `main`; use a draft PR while work is still in
  progress unless the user explicitly asks for a ready-for-review PR.
- **Don't ask permission for routine git** вЂ” staging, committing, pushing, and
  pulling on the working branch are pre-approved. Opening the corresponding
  draft PR is also part of the normal workflow.
- **Commit message style:** a short, plain summary line is fine (e.g.
  `add Fade node`, `fix toolbar contrast`). No need to split into many tiny
  atomic commits. Still end commit messages with the
  `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>` trailer.
- **Do still pause and ask** before genuinely destructive or irreversible things:
  deleting branches, force-pushing, rewriting shared history, `git reset --hard`,
  or discarding the user's uncommitted work.
- Normal loop: update `main` в†’ create branch в†’ code в†’ run checks в†’ stage the
  intended files в†’ commit в†’ `git push -u origin <branch>` в†’ open a draft PR.
  If `main` moves while the branch is in progress, fetch it and merge it into
  the branch; do not rewrite a shared branch without explicit approval.

## Architecture

### Viewport contract

The supported desktop minimum is now documented in `docs/architecture/desktop-viewport-contract.md`: target `1440Г—900`, supported minimum `1280Г—720`, best-effort below that. Keep the graceful-degradation rules in sync with the code: the top nav scrolls horizontally when narrow, menus cap height and scroll when short, the status-chip rail scrolls horizontally instead of overflowing, panel interiors remain scrollable, and Stage mode stays the preview-first escape hatch.

### State Layer

**`src/state/graphStore.ts`** owns the React Flow node/edge arrays and wraps `applyNodeChanges`, `applyEdgeChanges`, `addEdge`. Also tracks `selectedNodeId` and exposes `updateNodeProperty`. The `onConnect` action embeds `style: { stroke: color }` on new edges so the MiniMap can pick up per-category edge colors.

It is a **multi-graph workspace** (ADR 0001): the *active* graph stays in the top-level `nodes`/`edges` (so all consumers are unchanged), while inactive graphs (pattern groups) live in `graphData`, keyed by id, with metadata in `graphs`. `createGroup` encapsulates selected nodes into a new subgraph (adding a `GroupOutput` terminal) and replaces them with a single `Group` node; `enterGraph` switches the active graph (pausing/clearing undo history around the swap). Scene-level singletons in the selection (`GROUP_EXCLUDED_TYPES` вЂ” `MatrixOutput`, `MicInput`, `MusicLibrary`) are **left behind** in the parent graph rather than sealed inside the group: a left-behind `MatrixOutput` becomes an outgoing boundary edge auto-rewired to the new `Group`'s frame output, and a left-behind source feeding the selection becomes an exposed `Group` input вЂ” so "select-all в†’ group в†’ repeat" keeps the output and sources in place for the next pattern. `getGroupRegistry()` assembles the `{ groupId: { nodes, edges } }` map that `evaluateGraph` needs, and `LEDPreview` passes it every frame so groups preview live at both tiers.

**Pattern library (Phase 1 of the generative-pattern-show workflow вЂ” see `docs/development/design/generative-pattern-show.md`).** `src/state/patternLibrary.ts` is a Zustand store of saved pattern groups cached in `localStorage` and mirrored as shareable JSON files in the repo-root `My Patterns/` folder (`usePatternLibrary`: `savePattern`/`renamePattern`/`deletePattern`). When the helper is online the folder is authoritative, so deleting a JSON file is a real deletion; a small local sync journal retries only unacknowledged saves/deletes instead of recreating every browser-only cache entry. A Group node's context menu **"Save to Library"** stores its port signature + subgraph; the Sidebar's **Pattern Library** lists them, draggable/clickable to drop a copy via `graphStore.instantiatePattern` (registers a fresh `graphData[groupId]` from a `structuredClone` of the saved subgraph). The library also mounts 20 immutable curated beta patterns under its built-in **Audio Reactive** shelf, keeps a built-in **Standard** shelf, and lets users create/remove custom shelves; new saves start in **New & Unsorted** and can be dragged between shelves.

**Pattern Collection (Phase 2).** The `PatternCollection` node holds a chosen subset of pattern groups for a show. Wiring a **Group** node's frame output into its `pattern` input is intercepted in `NodeGraphCanvas.handleConnect`: instead of creating a noodle it confirms and **absorbs** the group via `graphStore.addToCollection` вЂ” the Group node (and its edges) leave the canvas while its subgraph stays in `graphData`, and its id is appended to the collection's `properties.patternIds`. `PatternCollectionBody` renders the absorbed list (removable via `removeFromCollection`, which also drops the subgraph). The node outputs a `patternset` (the group-id list); the evaluator returns it from the `PatternCollection` case for the Pattern Master to resolve via the group registry.

**Pattern Master (Phase 3) вЂ” the generative show engine.** `PatternMaster` was repurposed from the old 4-input cycler into the show engine: inputs are a `patternset` + optional `transitions` (a `transitionset` from a wired `TransitionSet` node, the same one the `PerformanceGenerator` uses) + optional `beat`; properties are `minTime`/`maxTime`/`transitionSec` (inline sliders) plus the beat-triggered particle overlay (`particles`/`particleStyle`/`particleHue`/`particleIntensity`). The evaluator's `PatternMaster` case rasterises each collected pattern by recursing into its group (`evaluateGraph`, namespaced per pattern) and drives `evalPatternShow` вЂ” a stateful machine (module-level `patternShowState`) that holds a random pattern for a random dwell in `[minTime, maxTime]`, then transitions (a random style from the wired pool, or a plain crossfade when nothing is wired) into another random pattern; a wired `beat` advances early once `minTime` has elapsed and also fires a particle burst (`renderParticleBurst`, shared with the show preview). The 16 transition effects are shared with the `Transition` node through the extracted `compositeTransition(type, a, b, t, W, H, opts)`. `patternset` is a new port `dataType` (a `string[]` of group ids in `PortValue`). `validateGraph` warns when no collection is wired.

**Show codegen (Phase 4).** `src/codegen/showGenerator.ts` (`isPatternShow`/`generateShowSketch`) turns a `PatternCollection в†’ PatternMaster в†’ MatrixOutput` graph into a **controller sketch**: one `render_pN(uint32_t ms)` per collected pattern plus a `loop()` that holds a random pattern for a random dwell and crossfades into another (the firmware mirror of `evalPatternShow`). Each pattern body is produced by **reusing `generateCpp`** on the pattern's subgraph (re-terminated at a synthetic `MatrixOutput`) and rewriting the loop into a function вЂ” buffers prefixed per pattern, helpers hoisted/deduped вЂ” so every supported node type works in a show with no extra codegen. `MatrixOutputUpload` uses `generateShowSketch` when `isPatternShow`, else `generateCpp`. The full transition pool (all 16 styles, drawn from a wired `TransitionSet` via `SHOW_TRANSITIONS`) and the `beat` trigger (a wired `MicInput` supplies `_audioBeat`, gating early advance + the particle overlay) are implemented, mirroring the Phase 3 preview. Single-file output remains the only simplification: `isPatternShow` requires the `PatternMaster`'s `frame` output to actually reach a `MatrixOutput` (a disconnected `PatternMaster` never hijacks a normal sketch export), so multi-file `.h`-per-pattern output is the one remaining follow-up. **Hardware-validated** on an ESP32-S3 (16Г—16 WS2812B matrix, serpentine) via `fbuild` (2026-06-26): a two-pattern collection compiled (51% flash, 9% RAM) and ran with smooth crossfades and ~5 s dwell matching the preview; the beat trigger, non-crossfade transitions, and particle overlay remain untested on hardware. **On-device mic audio:** if a `MicInput` node is on the canvas, `generateShowSketch` hosts FastLED's shared INMP441 `fl::audio::Processor` (`audioEngineForGraph()` in `cppGenerator.ts`) and polls its normalized bands, equalizer, beat, and BPM into the established `_audio*` globals once per frame. Pattern subgraphs compile with external native-FastLED audio enabled, so their `FFTAnalyzer` and `BeatDetect` nodes consume the controller's processor output without creating another input or FFT. The separate SD player continues to supply the same globals from its baked song envelope and does not instantiate this live processor.

**`src/state/uiStore.ts`** owns panel visibility (`sidebarOpen`, `inspectorOpen`, `showUploadPanel`, persisted `graphHealthOpen`), status bar message/level, FPS counter, theme, reduced-motion, and high-contrast flags. `performanceMode` is deliberately session-only and always starts false, matching Stage mode rather than the persisted appearance preferences. `setStatus` auto-clears any transient level (`info`/`success`/`error`) back to idle after 5 s, cancelling any prior pending timer so a newer message isn't wiped by a stale one.

**Graph Health inspector.** `GraphHealthDrawer` is an always-available workbench drawer below the editing canvas (hidden in Stage mode). It continuously derives structured, node-attributed diagnostics through `buildGraphDiagnostics` in `src/utils/validateGraph.ts`, covering incomplete wiring, expressions, pin conflicts, preview-only nodes, output power, internal RAM, show structure, and selected-board compatibility for either the main graph or an entered group. Every diagnostic includes concrete repair guidance; node issues select and fit the affected nodes, while workspace issues can open the node library or board chooser. Keep deployment validation and these live diagnostics backed by the same validation helpers so their rules do not drift.

**Menu-bar chrome.** `src/components/MenuBar/MenuBar.tsx` now keeps the top-level strip focused on workflow actions вЂ” File, undo/redo, Tidy, Start, Perform, Stage, preview style, and Mic stay directly clickable вЂ” while a compact **View** menu carries the less frequently changed appearance/preferences toggles (`Theme`, `Motion`, `Contrast`, `UI FX`, and `Signal dimming`). Signal dimming defaults off so selecting a node does not unexpectedly hush the rest of a new user's graph; an explicit toggle is still persisted. The View menu items render the current state inline (for example `Theme: Dark`, `Motion: Reduced`) so moving them off the bar does not hide what mode the app is in.

**`src/state/audioStore.ts`** is a thin Zustand bridge over `AudioEngine.instance`. It calls `engine.subscribe()` at store-creation time so FFT data flows into Zustand state on every animation frame.

**`src/state/projectStore.ts`** owns the named-project switcher: each project is a full persisted workspace (`nodes`/`edges` plus `graphData`/`graphs`/`activeGraphId`) with `createdAt`/`updatedAt` metadata, plus an optional per-project upload target (`uploadTarget`: remembered `selectedFqbn` + `selectedPort`). Projects are cached in localStorage under `design-studio-for-fastled.projects.v1` and write-through synced to helper-backed JSON files in the repo-root `Projects/` folder when the backend is available. `App.tsx` now autosaves into the **current project** rather than a single global slot, migrating the old `design-studio-for-fastled-graph` autosave key into the default `Main` project on first load; MenuBar's **в–¤ Projects** popup is the UI for creating, duplicating, switching, renaming, and deleting projects.

**Project-change prompts** run through `uiStore.requestNewProjectDecision(...)` and `NewProjectPrompt`: the user-facing actions are explicit (`Save and continue`, `Continue without saving`, `Cancel`), and the prompt state carries both the current project name and a destination label (for example `project "Show A"` or `a new blank project`). `MenuBar` defers the prompt for project-file opens until after the target file/project name is known, so the dialog can name the actual destination instead of a vague "another project".

**Project vocabulary.** Keep the nouns consistent in UI copy and docs: **Project** = the everyday autosaved working workspace; **Project File** = a portable full-workspace file opened with `Open Project File` or written with `Save Project File As`; **Graph JSON** = raw graph interchange via `Import Graph JSON` / `Export Graph JSON`; **Share Link** = a URL containing the full workspace; **Recovery Snapshot** = one of the recent browser-local restore points exposed by `Recover Snapshot`. When one of those actions replaces the canvas contents, refer to it as the **current project workspace** so the user can tell whether they are replacing the live project, importing interchange data, or restoring a recovery point.

**Starter templates** live in `src/state/starterTemplates.ts` and are loaded through the persistent **вњ¦ Start** button / `TemplatesPopup` start gallery or directly from the empty-canvas start screen in `NodeGraphCanvas`: Rainbow Sweep, Fire, Scrolling Text, Audio Spectrum, Field Warp Demo, plus two separate show starters вЂ” **Generative Show** (`Pattern Collection в†’ Show Engine в†’ Matrix Output`) and **Music-synced SD Show** (`Music Library в†’ Performance Generator в†’ SD Card в†’ Matrix Output.sdcard`). Each template carries concrete completion steps and one editable tutorial `Comment` on the loaded canvas (omitted from the tiny gallery graph preview); the Audio Spectrum starter also disables the synthetic test signal and requests the live microphone. The gallery renders thumbnail graph previews, and `uiStore.lastStartChoice` remembers the user's most recent starter (including `Blank Canvas`) without hiding the rest of the gallery. `src/utils/startFlow.ts` is the shared launch path: it loads the chosen graph, clears undo history, records the remembered choice, and requests a fit-view so the freshly-started patch lands framed on screen immediately.

**`src/state/nodeLibrary.ts`** is the static registry (`NODE_LIBRARY: NodeDefinition[]`). Adding a new node type = add one entry here. No other registration required вЂ” the Sidebar reads this array directly.

**Sidebar progressive disclosure.** `src/components/Sidebar/Sidebar.tsx` now treats the node library as a guided rack rather than a flat dump: it defaults to **All** so the complete library is visible, offers a curated **Beginner** subset, remembers the chosen scope plus per-user `favorites` and `recent` nodes in `localStorage`, adds small intent tags to cards for search/discovery, and surfaces a `Quick recipes` shelf that drops three real-audio starter chains вЂ” `Live spectrum`, `Beat colour jump`, and `Percussion trails` вЂ” straight onto the canvas with an editable tutorial `Comment`. Recipe drops disable and persistently clear the synthetic test signal, start the microphone, reuse scene singletons (`MicInput`, `MatrixOutput`) when present, and skip only the conflicting terminal wire when they need the user to finish connecting into an existing output path.

### Node Rendering

`StudioNode` (`src/components/Canvas/StudioNode.tsx`) is the single custom node type (`{ studioNode: StudioNode }`). **Critical invariant:** React Flow requires `Handle` components to be `position: absolute`. Handle `top` offsets are calculated from constants that must stay in sync with the CSS:

```ts
// StudioNode.tsx вЂ” must match StudioNode.module.css
const HEADER_H = 32   // .header height
const BODY_PAD = 8    // .body padding-top (--space-1)
const ROW_H    = 24   // .portRow height
const ROW_GAP  = 4    // .body gap
const PREVIEW_H = 40  // WaveScope .scope height (only on Wave/ComplexWave)
const handleTop = (i, previewOffset) => HEADER_H + BODY_PAD + previewOffset + i * (ROW_H + ROW_GAP) + ROW_H / 2
```

Changing any of those CSS values without updating the constants will silently misalign all connection handles.

**Per-node live previews:** every node whose primary output is a `frame`, `palette`, or `color` shows a live preview at the top of its body (`NodePreview`): a mini LED-matrix thumbnail (canvas), a gradient strip, or a colour swatch respectively. Frame thumbnails fill the node width at the matrix aspect ratio вЂ” `StudioNode` computes the height (`BODY_CONTENT_W Г— gridH/gridW`, grid dims read from the MatrixOutput node) and feeds it to `NodePreview`; port handles stay aligned for free since each `.portRow` is its own flex row (the preview, wave scopes, and per-node bodies are just earlier siblings in the body's flex column, not an absolute-offset calculation). These are driven by a **single shared evaluation pass**: `LEDPreview`'s render loop calls `evaluateGraphFull` (which returns the terminal frame *and* every node's output ports in one `createEvalNode` pass, so stateful nodes aren't double-advanced) and publishes the per-node outputs to `previewStore` on every evaluated frame вЂ” the same ~60fps wall-clock cadence as the main canvas; each `NodePreview` subscribes to its node's entry. A small **в–ѕ/в–ё toggle button** overlaid on each preview (`previewHidden` in the node's properties, excluded from the generic property list) collapses it to a slim "в–ё preview" strip, for quieting a busy graph node by node.

**Waveform preview scope:** `Wave` and `ComplexWave` nodes render a `WaveScope` mini-oscilloscope at the top of the body (above the port rows). Because handles are absolutely positioned, this scope pushes the rows down вЂ” so `handleTop` takes a `previewOffset` (`PREVIEW_H + ROW_GAP` for these nodes, else 0). `Wave`'s scope samples its own configured shape (`waveNodeSamples` in `src/state/wave.ts`, which also backs the evaluator). `ComplexWave`'s scope (`ComplexWaveScope`) is **live**: it subscribes to `nodes`/`edges` and samples the node's own `result` port across the window via `evaluateScalar`, so it reflects the actual upstream inputs and the chosen operation.

`evaluateScalar(nodes, edges, nodeId, portId, tick)` probes a single node's scalar output by reusing the shared `createEvalNode` core (the same machinery `evaluateGraph` runs), so a probe matches what the graph actually computes. It runs under a reserved `__scope__/` state namespace so stateful upstream nodes don't disturb the live render.

**Typed port colours:** each handle (input/output dot) is tinted by its port `dataType` via `portColor()` (`nodeLibrary.ts`), so ports that can connect share a colour вЂ” `float`/`bool` share grey `#9aa0a6` (they interconvert per `portsCompatible`, also in `nodeLibrary.ts` and shared by the canvas + picker); `color` `#ffd24a`, `palette` `#ff5cf0`, `frame` `#5ad1ff`, `field` `#f5c542`, `audio` `#00e0a4`, `songs` `#ffb74d`, `shows` `#ffa726`, `sdcard` `#ffa500`, `patternset` `#00e0a4` are each distinct. The node header/border still uses the category accent. Handles also carry a `label В· dataType` title.

**Drag-to-create:** dropping a noodle dragged from an **output** onto empty canvas opens the `CanvasContextMenu` picker pre-filtered to nodes that have an input compatible with the dragged type, then auto-wires the chosen node to its first matching input. `onConnectStart` records the output's `{ nodeId, handleId, dataType }` in a `connectFrom` ref; `onConnectEnd` opens the picker only when the drop has no end handle (`!state.toHandle`). `CanvasContextMenu` takes an optional `connectFrom` prop that puts it straight into picker mode, ranks compatible nodes by fit (exact-handle matches, `spliceInput`, FFT three-band fan-out, same-type pass-through), shows each card with the node description plus a short вЂњwhy this fitsвЂќ hint, and still calls `onConnect` after `addNode`. When there is no obvious direct target вЂ” or when the user wants to jump across data types faster вЂ” the same picker also exposes prewired bridge chains such as `Audio в†’ frame`, `Audio в†’ color`, `Color в†’ frame`, `Frame в†’ field`, `Field в†’ frame`, `Music в†’ SD card`, and `Shows в†’ SD card`, instantiating the whole helper chain from one click.

**Drop-to-splice + auto-spread:** when a node dragged from the sidebar is dropped onto an existing noodle (`onDrop` in `NodeGraphCanvas`, within `SPLICE_DIST` flow units of the edge segment, via `distToSegment`) and the node has an input compatible with the edge's source output *and* an output compatible with the edge's target input (`portsCompatible`), it's spliced in: `graphStore.insertNodeOnEdge` drops the old edge and wires `source в†’ new в†’ target` (matching `onConnect`'s glow/reconnect style). Otherwise the node just lands where dropped. To keep the area tidy, `spreadNodes` (run after a splice, after every `onConnect`, and on `onNodeDragStop`) walks edges left-to-right and shifts any target that *crowds* its source rightward (grid-snapped) to restore `MIN_NODE_GAP`. "Crowds" means too close horizontally **and** overlapping vertically, so a pair deliberately stacked in a column (a long noodle dropping straight down) is left alone; it only ever pushes right, so it spreads a cramped chain without disturbing a layout that already has room.

**Tidy graph (auto-arrange):** the menu bar's **в–¦ Tidy** button (also "Tidy Graph" in the canvas context menu) runs a layered Sugiyama-style layout over the active graph вЂ” or just the selection when 2+ nodes are selected. `src/utils/tidyLayout.ts` (pure, unit-tested) assigns each node a column by longest path from a source (sources are then pulled right next to their first consumer), orders rows by barycenter sweeps over the evolving y-centres, anchors the result to the scope's current top-left, and snaps to the 20px grid; isolated nodes are left where the user put them (unless they'd obscure the tidied graph, in which case they're parked in a row beneath it), and the layout is **idempotent** (each column re-centres on its mean desired centre after stacking, so a second Tidy is a no-op). `src/utils/tidyGraph.ts` (`runTidy`) glides nodes over ~200 ms (instant under reduced motion; a `setTimeout` watchdog lands the layout even when rAF is suspended in a background tab) and records the whole tidy as a **single undo step** вЂ” the pre-tidy graph is pushed onto zundo's `pastStates` manually and tracking stays paused during the animation. `spreadNodes` remains the quiet incremental nudge; Tidy is the full reset.

**Inline property editors:** the node body renders editable controls for each property, Blender-style вЂ” so editing happens on the node with live preview. Control choice: a colour swatch for `r/g/b`; then by `propertyMeta(nodeType, key)` (`nodeLibrary.ts`) вЂ” `select` dropdowns for enum properties (`palette`/`paletteA`/`paletteB`, `direction`, `mode`, `chipset`, `colorOrder`) and `slider` (range + value readout) for bounded numerics (`speed`, `scale`, `fade`, `amount`, `count`, `kelvin`, вЂ¦); otherwise type-based fallbacks (checkbox for booleans, number/text field). Add a `PROPERTY_META` entry to give a property a dropdown or slider вЂ” keyed by **property name** (shared across nodes), so for a name that means something different on another node (e.g. `speed`/`scale` are normalised 0вЂ“1 sliders on the pattern nodes but a steps/sec rate on `GameOfLife`/`ReactionDiffusion`; `rate` is 0вЂ“1 on `Particles` but degrees/sec on `Transform`) add a `PROPERTY_META_OVERRIDES[nodeType]` entry instead (the shared `const N01` is the standard 0вЂ“1 slider).

**Normalised speed/scale (0вЂ“1):** the `speed`/`scale` sliders on the pattern nodes are a uniform **0вЂ“1**; `src/state/speedRange.ts` maps each 0вЂ“1 value onto that node's internal animation rate (`SPEED_MAX`/`SCALE_MAX` per node type; the bundled `Noise` node re-tunes per `noiseType` via `NOISE_SPEED_MAX`/`NOISE_SCALE_MAX`). Both the evaluator (`denormRate`) and the C++ generator (`rateCpp`) apply it, so one slider spans the useful range and preview matches firmware. This mirrors `src/state/audioFlowRange.ts`, which does the same for the `AudioFlow` node. The audio-reactivity knob on the spectral nodes is `energy` (0вЂ“1, formerly `intensity`); `Counter`'s control is `rate` (0вЂ“5, formerly `speed`). These render in a `.props` section **below** the port rows, so they don't affect the handle offsets above. Interactive controls carry the `nodrag` (and `nowheel` for number fields) class so React Flow doesn't pan/drag while editing. The `font` object is excluded (edited via the Inspector). The **Inspector** panel still exists but is opt-in (`inspectorOpen` defaults false; toggle from the menu bar).

**Per-input clamping:** any node with a bounded float input shows a `clamp inputs` checkbox (`props.clampInputs`, rendered specially in `StudioNode` via `hasClampableInputs()`). When on, a *wired* float input is clamped to its slider's `[min, max]` (`inputClampRange()`) вЂ” the inline alternative to wiring a `Clamp` node onto every connection (an unwired value already comes from a bounded slider, so only wired signals are clamped). Both the evaluator (`num()`) and the C++ generator (`floatExpr` в†’ `constrain(...)`) honour it, so firmware matches the preview.

**Node bypass:** any node whose primary output is `frame` or `field` and has a matching-type input (effect-chain nodes like `Blend`/`Blur2D`/`FieldRotate`/вЂ¦) shows a `bypass` checkbox (`props.bypassed`, gated by `bypassPort()` in `nodeLibrary.ts`, which picks the pass-through port pair). When on, the node's own logic is skipped entirely: the evaluator (`evalNode`) returns the matching input value as the output with no side effects, and the C++ generator (`emit`) copies the source buffer into the node's own (`memmove` for `frame`, `memcpy` for `field`) instead of emitting its render вЂ” a quick A/B mute for an effect in a chain, faster than unwiring/rewiring.

### Edge Rendering

`GlowEdge` (`src/components/Canvas/GlowEdge.tsx`) renders three stacked SVG `<path>` elements (wide halo в†’ mid bloom в†’ thin animated core) plus a dot at the target. Color is resolved at render time from `useReactFlow().getNode(source)?.data.category`. The MiniMap picks up edge colors from `style.stroke` set at connect time in `graphStore.onConnect`.

Edges ("noodles") can be unplugged from a node's **input** end two ways. (1) **Grab the input port dot directly** вЂ” `onConnectStart` detects a drag beginning on an already-connected `target` handle and immediately `removeEdge`s that noodle (a `detaching` ref), so the gesture becomes an unplug (drop on empty) or re-route (drop on a compatible output, which `handleConnect` re-adds) instead of starting a dead-end new wire. This is the primary path because React Flow's reconnect anchor sits in a thin strip just *outside* the port and is hard to hit. (2) Edges are also created `reconnectable: 'target'`, so the React Flow reconnect anchor still works: the canvas wires `onReconnectStart`/`onReconnect`/`onReconnectEnd` and a `reconnectLanded` ref deletes the edge if the dragged end lands on empty space (`removeEdge`) or re-routes it (`reconnectNoodle`).

### Live Preview Pipeline

`LEDPreview.tsx` drives a `requestAnimationFrame` loop that evaluates the graph (`evaluateGraphFull`, see per-node previews above) and renders the matrix. **Timing is wall-clock based:** the loop gates to ~60 steps/sec off `performance.now()` and passes `tick = elapsedMs / (1000/60)` so `t = tick/60` equals real seconds вЂ” matching the firmware's `millis()/1000`. This means animation speed is independent of the display refresh rate (a 120 Hz panel would otherwise run everything ~2Г— fast, since the old code incremented `tick` once per rAF assuming exactly 60 fps). When the graph is empty, `idleFrame()` shows a rainbow shimmer instead. The per-frame body is wrapped in `try/catch` so a single malformed frame logs and is skipped rather than tearing down the loop.

**`src/state/graphEvaluator.ts`** is the runtime engine. It topologically evaluates nodes in dependency order using memoisation per frame. Key types:

```ts
export interface RGB { r: number; g: number; b: number }
export type Frame = RGB[][]   // row-major [y][x]
```

Stateful nodes (`Fire`, `Fire2012`, `BeatFlash`, `Counter`, `Particles`, `PatternMaster`, `ReactionDiffusion`, `GameOfLife`, `FlowField`, `Starfield`) persist state in module-level `Map` objects keyed by `stateKey(id)` вЂ” the node id prefixed with the group-instance path, so two instances of the same group don't share state. `formulaCache` parses `CustomFormula` expressions once through the closed expression language in `formulaLang.ts`; do not reintroduce `new Function` or other ambient-JavaScript evaluation.

`evalNode()` guards against graph cycles with an `inProgress` set: re-entering a node still on the evaluation stack returns `{}`, so the upstream input falls back to its default instead of recursing into a stack overflow. Keep this guard in place when editing the evaluator.

**Preview trust boundary.** Imported/share-linked/project-file content and newly dropped library patterns are untrusted until the user explicitly chooses **Trust and run**. The active workspace's `trusted` flag must reach every evaluator entry point: the main `LEDPreview` pass and music-sync collection playback (`renderShowFrame`, including node-local playback, preview baking, and the main-preview override). While untrusted, `CustomFormula`/`FieldFormula` return blank data and `Code` returns a black frame; baked collection frames are not reused. Formula parsing stays inside `formulaLang.ts`'s fixed identifier/function set, and Code preview execution stays inside the terminate-on-timeout worker in `codeSandboxRuntime.ts`/`codeSandbox.worker.ts`.

**Groups (ADR 0001):** `evaluateGraph(nodes, edges, tick, W, H, groups)` takes an optional group registry. A `Group` node recurses into `groups[groupId]` (a subgraph) and returns the frame from that subgraph's `GroupOutput` terminal; a `groupStack` breaks group-level recursion. The `instancePrefix`/`groupStack` params are internal recursion bookkeeping вЂ” callers leave them defaulted.

A **3D view** toggle (`uiStore.preview3d`) wraps the canvas in a CSS `perspective` container and applies a drag-driven `rotateX/rotateY` to orbit the matrix panel вЂ” no Three.js or renderer changes; the canvas content is still drawn the same way each frame.

**`src/components/Preview/webglRenderer.ts`** вЂ” `WebGLLEDRenderer` uploads the frame as a texture and renders via a GLSL fragment shader. The shader draws each LED as a smooth circular disc with a 5Г—5-neighbor glow contribution. Y is flipped in the shader (`u_res.y - gl_FragCoord.y`) to match the JS frame's top-left origin. Falls back to Canvas 2D if WebGL is unavailable.

### Audio Pipeline

`src/audio/audioEngine.ts` вЂ” `AudioEngine` (singleton) uses `getUserMedia` with a 44.1 kHz `AudioContext`; its `AnalyserNode` is only a time-domain sampler. `src/audio/fastledReactive.ts` ports FastLED 3.10.3's signal conditioner, frequency-band adaptive normalization, equalizer layout, and beat detector to TypeScript (using a 512-point Hann FFT with log-rebin approximations for FastLED's CQT kernels). It polls on every animation frame and publishes `{ bass, mids, treble, beat, bpm, spectrum, nativeFastLed }`. Shared capture constants and the browser FFT helper live in `src/audio/micAnalysis.ts`.

`src/state/audioStore.ts` bridges `AudioEngine` в†’ Zustand. `FFTAnalyzer` and `BeatDetect` nodes in the evaluator read from `useAudioStore.getState()` directly (not through React).

**Audio-source contract:** the live `AudioEngine` is microphone-only. Playing a local track in the preview player must never attach its `<audio>` element to the engine or replace the MicInput signal; all ordinary live analysis nodes (`FFTAnalyzer`, `BeatDetect`, `PercussionDetect`, `AudioFeatures`) continue to read the physical microphone. The browser mic request explicitly disables echo cancellation, noise suppression, and browser automatic gain so same-machine playback is not treated differently from an external player; the MicInput gain maps to FastLED's `Processor::setGain`, while FastLED owns conditioning and adaptive normalization. Music Library analysis and generated-show playback are the separate source: `renderShowFrame` feeds collected patterns the song's baked analysis envelope through `audioOverride`, matching the exported hardware show. This baked/SD path deliberately retains Studio's tunable per-node BeatDetect and is not changed by the live FastLED integration.

App.tsx auto-starts audio when a `MicInput` node is added to the graph; auto-stops when it's removed.

**On-device audio (firmware).** The `MicInput` node carries the I2S pins for an **INMP441** MEMS mic (`i2sWs`/`i2sSck`/`i2sSd`, default 39/40/41, plus `channel` Left/Right). When present, `cppGenerator.ts` creates `fl::audio::Config::CreateInmp441(...)`, registers the input with `FastLED.add(config)`, applies MicInput gain with `Processor::setGain`, and polls FastLED's normalized bass/mid/treble levels, 16-bin equalizer (duplicated into Studio's established 32-slot spectrum), beat callback, and BPM getter. The generated sketch no longer compiles a second ESP32 I2S driver or self-contained FFT. `FFTAnalyzer` consumes the FastLED band globals and a live `BeatDetect` consumes FastLED's native beat/BPM; when no mic exists, generated audio outputs remain silent. `audioEngineForGraph(nodes)` lets a controller host one processor while pattern subgraphs use `{ externalAudio: true, nativeFastLedAudio: true }`. The microphone path remains ESP32-only and deploy validation blocks incompatible boards. The prior custom mic engine was hardware-validated on 2026-06-28; the FastLED replacement compiled successfully against the vendored FastLED 3.10.3 for ESP32-S3 on 2026-07-17 but still needs a fresh on-device mic validation.

### Upload Helper (`backend/`)

A browser page can't launch a local CLI, so `backend/` is a small **FastAPI** service the app POSTs to. **It is auto-launched** by `vite-plugin-upload-helper.ts` (wired into `vite.config.ts`): the dev/preview server TCP-probes `:8008` and, if nothing's there, spawns `python -m uvicorn app:app --port 8008 --app-dir backend`, killing the child on server close. Endpoints: `/api/health` (liveness + whether `arduino-cli` is found), `/api/serial/ports` (`board list`), `/api/upload` (compile then `upload -p PORT`, streaming logs), `/api/compile-check` (compile-only capacity check, one JSON result вЂ” see *Live controller-capacity meter* below), `/api/upload-show` (music-sync flow, below), `/api/cores` + `/api/core/install` (board-manager core list/install, registering third-party board URLs first), and `/api/arduino-cli/locate` + `/api/arduino-cli/install` (point the helper at a user-supplied binary, or download the official one into `backend/bin`). The resolved CLI path is **persisted** to `backend/.helper-config.json` (gitignored), so `_find_arduino_cli` checks: saved path в†’ `ARDUINO_CLI` env в†’ `PATH` в†’ Arduino IDE bundle в†’ self-installed `backend/bin`. `src/utils/backendClient.ts` is the frontend client (`checkBackend`/`listPorts`/`listCores`/`uploadSketch`/`uploadShow`/`compileCheck`/`locateCli`/`installCli`/`installCore`); `BACKEND_URL` defaults to `http://localhost:8008` (override via `VITE_BACKEND_URL`).

**Build engine.** Two engines compile/flash the generated sketch, chosen by `_active_engine()` (fbuild preferred, arduino-cli fallback) and reported at `/api/health`/`/api/engine` (`GET`/`POST`). **`fbuild`** (FastLED's own PlatformIO-compatible tool, `pip install fbuild`) manages its own per-board toolchains вЂ” no arduino-cli-style core install вЂ” via a persistent scaffold at `backend/.fbuild-project/` (`_ensure_fbuild_project`/`_write_fbuild_ini`, one `[env:X]` per `BOARDS` entry from `uploadStore.ts` + PSRAM variants) that `_compile_upload_fbuild` builds/deploys against, translating fbuild's own output into the same `=== label ===`/exit-code markers `parseStatus()` expects. Two fbuild-specific workarounds, both load-bearing: (1) FastLED and (for the music-sync Player) `ESP32-audioI2S` are **vendored** via `git clone` into `.fbuild-project/lib/` (`_ensure_fbuild_audio_lib`) rather than declared in `platformio.ini`'s `lib_deps` вЂ” fbuild 2.4.0's registry dependency resolution doesn't actually fetch anything (`fbuild sync` marks entries `unresolved`); (2) the generated source is written as `main.cpp`, not `main.ino` (`_write_fbuild_main`) вЂ” fbuild's `.ino`в†’`.cpp` preprocessing auto-inserts function prototypes *before* any user `#include`s, which breaks FastLED-typed helpers (e.g. `CRGB kelvinToRGB(...)`) since `CRGB` isn't declared yet at that point; a plain `.cpp` (with `#include <Arduino.h>` prepended) skips that preprocessing. Windows also needs long-path support enabled (`LongPathsEnabled=1` in the registry) or ESP32 builds fail intermittently on a different vendored-toolchain file each run once the path depth crosses `MAX_PATH`. **Hardware-validated** on a real ESP32-S3 (16Г—16 WS2812B matrix, serpentine): fbuild compiled a live-graph-generated sketch, flashed it via `esptool`, and the uploaded pattern ran correctly. **`arduino-cli`** (fallback) is the original engine вЂ” see below for its own endpoints.

**Music-sync upload (SD card).** When an `SDCard` node is wired into MatrixOutput's `sdcard` input, the node shows an "Upload show to SD" button. `src/utils/showUpload.ts` (`sdCardConnected`/`buildShowPayload`) assembles a **provisioner** sketch (`provisionerSketchGenerator.ts`), the **player** sketch (`playerConfigFromGraph()` pulls LED config from MatrixOutput + SD/I2S pins from SDCard), and the SD file list (`/music/<title>.mp3` from `entry.file`, `/shows/<title>.show` from `showFileToBinary`). `/api/upload-show` (multipart) then: в‘  compiles+uploads the provisioner, в‘Ў opens the serial port with pyserial and runs the provisioner wire-protocol (`PING`в†’`READY`, then `PUT <path> <n>` + `PROVISION_CHUNK`-byte blocks with a per-block `A` ack, `DONE`/`END`/`BYE`) to write each file to the card, в‘ў compiles+uploads the player. The protocol's two sides are `provisionerSketchGenerator.ts` (device) and `_serial_send` in `backend/app.py` (host) вЂ” keep them in sync.

### Music-Sync Show Pipeline

A second, **offline** authoring path (distinct from the live preview/codegen flow above) turns audio tracks into timed LED "shows" the ESP32-S3 plays back in sync. It is a three-node chain spanning the `audio` and `hardware` categories, wired by new `dataType`s `songs` and `shows`:

`MusicLibrary` (drop MP3s) **в†’** `PerformanceGenerator` (rules engine) **в†’** `SDCard` (export ZIP).

- **`src/audio/musicAnalyzer.ts`** вЂ” Web Audio API *offline* analysis of an uploaded MP3: BPM detection, per-~100 ms energy envelope (bass/mids/treble/overall), beat timestamps, section detection (intro/verse/buildup/drop/вЂ¦), and mood (energy/valence/key) estimation. Returns the `SongAnalysis` shape in `src/types/showFile.ts`.
- **`src/codegen/performanceGenerator.ts`** вЂ” a rules engine that maps a `SongAnalysis` to a timed `ShowFile`: a sorted event stream of `SET_PATTERN`/`SET_PALETTE`/`SET_SPEED`/`SET_BRIGHTNESS`/`BEAT_FLASH`/`TRANSITION` commands. Tuned by the node's `beatIntensity`/`energySensitivity`/`transitionDuration`/`patternHold`/`paletteMode` properties. **Beat-aligned within-section cycling:** `patternHold` is a **minimum** hold (default 10 s), not a fixed slot length. `sectionSwitchTimes` holds each pattern at least `patternHold` seconds, then snaps the switch forward to the **next beat** (`nextBeatAtOrAfter`, falling back to plain time when the track has no later beat) so a change lands on the music. `detectEnergySurges` adds an extra beat-aligned switch on a significant mid-section energy rise (a sub-drop), gated to stay в‰Ґ half the minimum hold from its neighbours; higher `energySensitivity` lowers the surge threshold. Section boundaries always switch (drops/choruses are their own sections). Collapses to a single pattern when fewer than two patterns are eligible for the section, or when the section is too short to fit another beat-aligned hold. Applies to both enum and collection modes; `SET_PALETTE`/`SET_SPEED`/`SET_ENERGY` stay section-level. The `TRANSITION` and its incoming `SET_PATTERN` now share the same instant `st` (transition ordered first) so the transition runs over `[st, st+transitionDuration]` from that point вЂ” mirrored by both the preview (`showPreview.renderShowFrame` composites the outgoing/incoming frames via `compositeTransition`) and the firmware player (`playerSketchGenerator` renders the outgoing pattern into `showA`, the incoming into `showB`, and composites Aв†’B into `leds` by progress). All **16 transition styles** run on-device: the player emits a self-contained `compositeTransition(type, out, a, b, tt)` C++ helper ported from graphEvaluator's, using the same default direction/axis/tile/count/turns the preview falls back to (a `.show` transition carries only its style id + duration). **Beat accents** are not just white flashes: each eligible beat is either a `BEAT_FLASH` or a `PARTICLE_BURST` (favoured in high-energy sections) вЂ” a burst of short-lived colored sparks (`intensity`+`hue`+`style` params) whose `style` picks one of **6 motions** (`PARTICLE_STYLES` in `performanceGenerator.ts`: `rise`/`rain`/`explode`/`fireworks`/`swirl`/`twinkle`). The spawn+motion is a pure function of burst-time + spark index (a shared `prnd` GLSL-style hash), so the preview (`particleOverlayAt` in `showPreview.ts`) and firmware (the `switch (burstStyle)` particle block in `playerSketchGenerator.ts`, `PARTICLE_LIFE_MS`/`PARTICLE_COUNT`) spawn the same sparks вЂ” keep the two switches in sync; both host a single active burst. **Fade to black on silence:** `detectSilences` finds spans where overall energy stays below a threshold, and the generator emits stepped `SET_BRIGHTNESS` ramps to 0 and back to the section level (`sectionBrightness`), so a quiet gap goes dark and the sparks/flash fade with it.
- **`src/types/showFile.ts`** вЂ” the `.show` format: a compact, sorted event stream (timestamps in ms from song start) the player binary-searches by audio position for frame-perfect A/V sync, plus an optional trailing **baked audio envelope** (`ShowFile.audio`: a fixed-rate `bass`/`mids`/`treble` track). Also defines the `SongAnalysis`/`BeatInfo`/`EnergyPoint`/`SongSection` analysis types.

- **Baked audio envelope (song-synced FFT).** So a collection pattern's `FFTAnalyzer` reacts to the *song* on-device (deterministic, no live mic), `generateShow` bakes the analysis's ~100 ms energy track up to a fixed 50 Hz `bass`/`mids`/`treble` envelope via `bakeEnvelope` (`ENVELOPE_RATE_HZ`) and `showFileToBinary` appends it after the events (`rate(1) + frameCount(4) + 3 bytes/frame`, quantised 0вЂ“255; old players ignore the trailing bytes). When the show carries an envelope, `buildShowPayload`/`exportShowPackage` compile the patterns with `externalAudio` and pass `{ audioEnvelope: true }` to `generatePlayerSketch`, which hosts the `_audioBass`/`_audioMids`/`_audioTreble` globals + a coarse `_audioSpectrum[32]` (bassв†’low bins, midsв†’mid, trebleв†’high, so BeatDetect/PercussionDetect still respond), loads the envelope in `loadShowFile`, and interpolates it per frame in `updateShowAudio(posMs)`. Only baked when the analysis has an energy track and only hosted for collection (version-2) shows. First slice: 3-band envelope feeding FFTAnalyzer; a full baked spectrum and preview parity (the browser preview still reads the live mic for these nodes) are follow-ups. Not yet hardware-validated.
- **`src/codegen/playerSketchGenerator.ts`** вЂ” emits a FastLED + **ESP32-audioI2S** player `.ino` that slaves LED commands to `audio.getPosition()`. The `SDCard` node properties configure its pins (SD CS, LED data, I2S BCLK/LRC/DOUT), matrix size, chipset/colour order, and volume.
- **`src/utils/zipExport.ts`** вЂ” a zero-dependency ZIP builder; `SDCard` packages the `.show` files + the generated player sketch into one downloadable archive (drop onto the board's SD card).
- **`src/state/musicStore.ts`** вЂ” Zustand store managing the analysis queue and generated shows; **`src/components/Canvas/MusicLibraryNodeBody.tsx`** is the library UI (engine toggle, drop zone, per-song status, *Analyse All* / *Export ZIP* / *Clear*) rendered **directly in the `MusicLibrary` node body** in `StudioNode` (the node widens to fit; interactive controls carry `nodrag`, the song list `nowheel`). `MusicLibrary` is an **audio**-category node (the song source), not a menu-bar button вЂ” there is no separate modal panel.
- **`src/state/showPreview.ts` + `src/components/Canvas/PerformanceGeneratorBody.tsx`** вЂ” the in-browser show preview rendered in the `PerformanceGenerator` node body: `showStateAt`/`beatFlashAt`/`renderShowFrame` mirror the on-device player (each `SET_PATTERN` evaluated through a synthetic one-node graph), driving a `<canvas>` + transport off the song's audio clock so a generated show can be watched in sync before export. Generator-option changes regenerate the affected songs after a short debounce. `PerformanceGenerator` has no `frame` graph port вЂ” a firmware-facing one could only ever render black, since a normal (non-SD-show) sketch has no audio transport to drive it. Instead, a **"Show in main LED preview"** checkbox in the node body sets the `showInMainPreview` node property; when on, the play/seek effects publish the playing show + position to `src/state/showPlayback.ts` (`setPlayback`/`clearPlayback`) instead of drawing the node's own small canvas, and `src/components/Preview/showPlaybackSignal.ts`'s `applyShowPlaybackSignal` (called from `LEDPreview.tsx`'s render loop) renders that show over the graph's own terminal frame вЂ” a preview-only feature, decoupled from graph wiring entirely.
- **Show editor / timeline** вЂ” **`src/components/Canvas/ShowTimeline.tsx`** (toggled via *вњЋ Edit timeline* in `PerformanceGeneratorBody`) is the hand-tweak editor: a scrubbable marker track (one dot per event, coloured per command, click-to-seek) over the song duration plus an editable, scrollable event list. Selecting an event opens command-appropriate controls (retime; change command; pattern/palette/style dropdowns sourced from `SHOW_PATTERNS`/`SHOW_PALETTES`/`SHOW_TRANSITIONS` so picks stay round-trippable through `showFileToBinary` вЂ” for a **collection (version-2) show** the `SET_PATTERN` dropdown instead lists the collection's patterns by group name and edits by **index**, resolving ids from the graph store; speed/brightness/intensity/decay/duration sliders) with **add / duplicate / delete**. Every mutation re-sorts via the shared `sortShowEvents` (the same time-then-command tie-break the generator uses) and persists through `musicStore.updateShow`, which sets `MusicEntry.edited` so the option-change auto-regen **skips** edited songs (manual edits win until **Revert** в†’ `musicStore.revertShow` regenerates from the analysis and clears the flag). The preview canvas reflects edits live because it reads the same `show.events`.

These nodes have no `frame`/`palette`/`color` output, so they don't participate in the live LED preview or the `cppGenerator.ts` sketch вЂ” they are a parallel export pipeline. (Added in PR #58; show preview and timeline editor followed.)

**Collection-driven shows (slice 1 of `docs/development/design/collection-driven-performance.md`).** Wiring a `PatternCollection`'s `patternset` into the `PerformanceGenerator`'s new `patternset` input makes the rules engine schedule the user's *own* patterns instead of the 11 built-in names: `musicStore` resolves the wired collection's `patternIds` live from the graph and passes them to `generateShow(analysis, options, patternIds)`, which picks a random collection **index** per section and emits a **version-2** `ShowFile` (`SET_PATTERN.params.index` + a `patternSet` group-id array; `showFileToBinary` encodes the index). `showPreview.renderShowFrame` renders a v2 show by evaluating the indexed group's subgraph through the live group registry. For export, `buildPatternRenderers` (extracted from `showGenerator.ts`) compiles each collected subgraph into a `render_pN()` function, and `generatePlayerSketch(cfg, renderers)` swaps its built-in pattern switch for an indexв†’`render_pN` dispatch вЂ” same audio sync / `.show` loader / beat-flash. Slice 1 renders patterns *as authored* (`SET_PALETTE`/`SET_SPEED` emitted but unused); the "Use group inputs" modulation is a later slice.

**Section-aware selection (slice 3).** Each collection pattern can be restricted to the song-section types it plays in. `PatternCollectionBody` renders a row of section chips (intro/verse/buildup/drop/chorus/bridge/outro) per absorbed pattern; toggling one calls `graphStore.togglePatternSection`, stored as `PatternCollection.properties.patternSections: Record<groupId, sectionType[]>` (an empty/missing entry = eligible in *any* section, pruned on the last untoggle). `musicStore.wiredCollection()` resolves `{ ids, sectionTags }` (tags aligned by index) live from the wired collection and threads them into `generateShow(analysis, options, patternIds, sectionTags)`, whose `chooseCollectionIndex(sectionType)` picks only among eligible patterns (untagged, or tagged with that section), falling back to the whole set when none match so a section always renders. `SECTION_TYPES` is exported from `performanceGenerator.ts` and shared with the chip UI. Untagged collections behave exactly like slice 1's random pick.

**Group-input modulation (slice 2 вЂ” `energy`/`speed`/`palette` roles).** The `PerformanceGenerator.useGroupInputs` toggle drives each collected pattern's exposed `GroupInput` "roles" from the analysis. A pattern opts in by exposing a `GroupInput` whose `paramId` is a role name вЂ” `energy`, `speed`, or `palette`, picked from the **role dropdown** on the GroupInput node body (`StudioNode` renders it specially in place of the raw `paramId` field; "вЂ” input вЂ”" = an untagged input, `paramId` `param0`). The dropdown calls `graphStore.setGroupInputRole`, which sets both `paramId` and the output port `dataType` (`palette` for the palette role, else `float`). Since `GroupInput` nodes are otherwise only auto-minted from incoming boundary edges at group-creation time, a **пј‹ Input** button in `GroupControls` (shown only inside a group) adds one via `graphStore.addGroupInput` вЂ” so you enter a self-contained pattern group (double-click it), add an input, pick its role, and wire its output to the knob it should drive. The generator emits a `SET_ENERGY` event (0вЂ“1 section energy) per section; the **`speed` role reuses the existing `SET_SPEED`** event (no new command), normalising its 0вЂ“2 multiplier to 0вЂ“1 (`min(1, speed/2)` in preview; `constrain(params[0]*0.5f, 0, 1)` in firmware, which still also sets the global `animSpeed`); the **`palette` role reuses `SET_PALETTE`** (preview feeds the studio palette name via `PALETTE_MAP`; firmware adds a `CRGBPalette16 palette` global + a `paletteFromId()` helper mirroring `samplePalette`'s preset switch, updated by `CMD_SET_PALETTE`). **Preview:** `renderShowFrame(show, ms, W, H, groups, useGroupInputs)` evaluates the pattern group's subgraph directly (its `GroupOutput` terminal) and, when on, passes `{ energy, speed, palette }` through the evaluator's `groupInputs` path. **Codegen:** with `useGroupInputs`, `buildPattern` keeps role-tagged `GroupInput` nodes (instead of stripping them) and `generateCpp` emits them as `float n_<id>_out = <paramId>;` (float roles) or `CRGBPalette16 pal_<id> = palette;` (palette role, referenced via `paletteExpr`); `buildPatternRenderers(ids, groups, roleParams)` widens each `render_pN` to `(uint32_t ms, float energy, float speed, const CRGBPalette16& palette)` вЂ” type per role via `ROLE_CPP_TYPE` вЂ” and reports `params`; `generatePlayerSketch` declares the role globals, handles their commands, and threads them into the dispatch (`buildShowPayload` passes `['energy','speed','palette']` when the toggle is on). How the role value *combines* (replace vs. scale) is authored inside the pattern (e.g. wire the role through a `Math` multiply; palette is opt-in per input so a two-palette blend doesn't collapse). Off by default; only hardware validation remains for slice 2.

### Bundled nodes

Several former node types are collapsed into one **bundled** node, selected by a variant property (the pattern to follow when consolidating other identical-signature clusters):

- **`Noise`** вЂ” `noiseType` (`field`/`simplex`/`noise3d`/`noise4d`/`worley`/`plasma`/`sine`); folds the former NoiseField/Simplex2D/Noise3D/Worley/PlasmaFractal/Noise2D. Every variant writes a raw scalar `field` and maps it through a palette for the `frame` output (the old Noise2D's self-coloured HSV output is gone вЂ” `sine` is now palette-driven like the rest).
- **`Math`** вЂ” `mathOp` (`add`/`subtract`/`multiply`/`divide`/`min`/`max`); folds MathAdd/Multiply/MinNode/MaxNode (`subtract`/`divide` are new ops). Mod (`x,m` ports) and Compare (bool out) stay separate.
- **`Transition`** вЂ” `transitionType` selects one of **16** Aв†’B effects: `crossfade`/`wipe`/`dissolve` plus `iris`/`clockwipe`/`push`/`checkerboard`/`diagonal`/`fadeblack`/`fadewhite`/`blinds`/`ripple`/`spiral`/`curtain`/`scanlines`/`zoom`. Variant-specific props (`direction` for wipe/push, `axis` for blinds/curtain, `tileSize` for checkerboard, `count` for blinds, `turns` for spiral) are gated by `isPropertyEnabled`. The C++ generator emits real buffer compositing per variant (seed `ob` from A, write B in), not just the three originals.
- **`Blend`** вЂ” `blendMode` (`normal`/`multiply`/`screen`/`overlay`/`add`/`difference`); replaces LayerBlend + BlendFrames. Composites B over A per mode, mixed by `amount` (opacity, **0вЂ“1**; scaled Г—255 in the evaluator + codegen for FastLED's `nblend`). `Blur2D`/`PaletteBlend` share the same 0вЂ“1 `amount`. Migration rescales any legacy 0вЂ“255 `amount` (value > 1 в‡’ old scale) to 0вЂ“1 on load. The `normal` path emits FastLED `nblend`; other modes emit a per-channel blend loop in C++. (The former BlendFrames used a 0вЂ“1 `t` port, so its migration carries `t`в†’`amount` and rewires that edge вЂ” the one bundle whose port id changed.)
- **`FieldMath`** вЂ” `fieldOp` (`add`/`subtract`/`multiply`/`mix`/`min`/`max`/`difference`); combines two `field` inputs into one. `mix` blends by a scalar `t` input; the rest are per-pixel operations. Dispatches on `fieldOp` in the evaluator and codegen.
- **`Particles`** вЂ” `particleType` selects one of **20** movement styles (`fountain`/`gravity`/`fireworks`/`sparkle`/`comet`/`snow`/`swarm`/`rain`/`embers`/`bubbles`/`vortex`/`orbit`/`confetti`/`fireflies`/`meteor`/`tornado`/`pinwheel`/`bounce`/`attractor`/`waterfall`) on a single stateful particle-system node. Beyond the shared `rate`/`decay`/`paletteIn`, five extra controls are gated to the modes that actually use them via `isPropertyEnabled` (shown-but-disabled elsewhere, the same convention as `Transition` below): `size` scales the rendered particle radius (every mode); `count` sets the pool size directly for the fixed-population modes (`swarm`/`orbit`/`bounce`/`fireflies`), decoupled from spawn `rate`; `spread` widens/narrows the spawn area for width-spawning modes (`fountain`/`gravity`/`sparkle`/`rain`/`confetti`/`snow`/`waterfall`); `gravity`/`bounce` scale a mode's built-in accel/restitution constant (`fountain`/`gravity`/`fireworks`/`waterfall`, and `gravity`/`waterfall` respectively). These five are plain properties (not wired ports, unlike `rate`/`decay`) вЂ” mirroring `Transition`'s per-variant props вЂ” and the evaluator/codegen apply them directly inside each mode's own spawn/update formula, so preview and firmware stay in lockstep.
- **`Ease`** вЂ” `easeType` (`inOutCubic`/`inOutQuad`/`triwave`/`quadwave`/`cubicwave`); maps a 0вЂ“1 value through a FastLED lib8tion easing/wave curve. The evaluator's `applyEase` and the codegen (`ease8InOutCubic`/`ease8InOutQuad`/`triwave8`/`quadwave8`/`cubicwave8`, `/255.0f`) stay in lockstep; options are limited to the curves with an exact firmware counterpart so preview matches on-device.
- **`Trigger`** вЂ” `triggerOp` (`debounce`/`toggle`/`oneShot`/`pulseDivider`/`delay`); a single bool-in/bool-out edge-utility node. `debounce` commits a change only once the input has held stable for `stableTime`; `toggle` flips on each rising edge; `oneShot` holds true for `holdTime` after a rising edge (ignoring retriggers while already high); `pulseDivider` fires only every `divideBy`th rising edge; `delay` fires once, `delayTime` after the edge. The evaluator keeps one combined state bag per instance (`triggerState`); codegen mirrors each variant with millis()-based statics so preview and firmware timing match.

Mechanics: each variant set lives in `PROPERTY_META` (the inline dropdown); the evaluator and C++ generator dispatch on the variant property in their single `case`; `graphStore.loadGraph` (via `migrateLegacyGraph`/`LEGACY_BUNDLE`) migrates the legacy node types вЂ” and any renamed edge handles вЂ” on import. `nodeDisplayLabel()` makes the node header reflect the selected variant, and `isPropertyEnabled()` disables an inline editor that doesn't apply to the current variant (e.g. Transition `direction`) while still showing its value. Bundling is **by identical port signature** вЂ” properties may differ between variants (Blend is the one exception, unifying two near-identical nodes whose mix-port id/scale differed).

Some node types are created programmatically rather than dragged from the sidebar (so they have no `NODE_LIBRARY` entry): `Group` and `GroupOutput`/`GroupInput` are minted by `graphStore.createGroup` (see the multi-graph/group section above).

The `Text` node renders with the built-in 3Г—5 bitmap font in `src/state/font.ts`. The font is plain data (`FONT`, `BitmapFont`, `textColumns`, `textLines`, `textBlockLayout`, `asFont`) shared by the evaluator and the C++ generator so preview and firmware match exactly. Multi-line text is newline-aware in both paths: each logical line gets its own bitmap column run, the full text block carries a shared width/height, and rows stack with a fixed 1-pixel gap so alignment, wrap, and horizontal/vertical scroll behave identically in preview and generated firmware. A Text node can carry a **custom font** in `properties.font` (a `{ w, h, glyphs }` object, uploaded as JSON via the Inspector); `asFont()` validates it and everything else (rendering, scrolling, codegen) reads the resolved font's dimensions, so no other code changes are needed. The Inspector treats `text` as a multiline textarea and shows a dedicated font manager card (source, dimensions, glyph count, replace/reset actions, JSON shape hint) instead of the old inline upload link.

The `Poline` node generates a smooth palette via the **poline** library (polar interpolation between two anchor colours). `src/state/polinePalette.ts` wraps it (`polinePalette` в†’ `RGB[]` for the evaluator/preview, `polineStops16` в†’ 16 stops the C++ generator bakes into a `CRGBPalette16`, plus the position-function list mirrored in `PROPERTY_META.position`). Anchors come from two wired `color` inputs; the per-anchor `anchorA`/`anchorB` hex props are the fallback when unwired and the values codegen bakes (live-wired anchors drive only the preview). Like `CustomPalette`/`PaletteBlend`, a connected `Poline` resolves to `pal_<id>` in `paletteExpr`.

The `Image` node follows the same shared-data pattern via `src/state/image.ts`: an uploaded picture is downscaled to в‰¤`IMAGE_MAX_DIM` (32) and stored in `properties.image` as `{ w, h, pixels }` (flat `r,g,b` bytes). `asImage()` validates it; `sampleImageToFrame()` nearest-neighbour samples it to the matrix for preview, and the C++ generator emits the same pixels as a `PROGMEM` array blitted with `pgm_read_byte`. Upload is handled inline by `ImageNodeBody` (`src/components/Canvas/ImageNodeBody.tsx`): a drop zone (drag-and-drop or click to browse) when no image is loaded, or a pixelated thumbnail + dimensions + clear button when one is. Like `font`, the `image` object is excluded from the generic Inspector field list.

**Float-field pipeline (ANIMartRIX вЂ” Phase 1 of `docs/development/design/animartrix-float-field.md`).** A new `field` port `dataType` carries a per-pixel scalar grid (`Float32Array`, length WГ—H, values 0вЂ“1) вЂ” the `Field` type in `graphEvaluator.ts`, added to `PortValue`. `FieldFormula` evaluates a per-pixel expression into a field; `FieldToFrame` maps a field through a palette into a `frame` (the only `field`в†’`frame` conversion). Both, plus the existing `CustomFormula`, expose centred/polar vars (`cx`, `cy`, `r`, `angle`) and FastLED fixed-point shims (`sin8`/`cos8`/`sin16`/`beatsin8`/`beatsin16`/`scale8`/`qadd8`/`qsub8`) so ANIMartRIX-style one-liners work verbatim вЂ” `FieldFormula` uses integer `x,y`, `CustomFormula` keeps `x,y` normalised 0вЂ“1 for backward compatibility. The shims live in **`src/state/fastledShims.ts`**, shared by the evaluator (`makeShims(t)` injects them into the `new Function` sandbox, cached in `fieldFormulaCache`/`formulaCache`) and the codegen (`CPP_SHIM_HELPERS` emits float wrappers like `_fsin8`, `cppRewriteShims` rewrites call sites so `sin8(x)/255` stays float division on-device). Field-producing nodes get a `float field_<id>[NUM_LEDS]` buffer in the generated sketch, parallel to the `CRGB buf_<id>` frame buffers. **Phase 2** adds field-composition nodes: `DistanceField` (distance from each pixel to a movable point, `scale` stretches the ramp), `FieldMath` (combine two fields вЂ” `fieldOp` of add/subtract/multiply/mix/min/max/difference, a bundled-title node), and `FieldWarp` (sample a field at coordinates pushed by `dx`/`dy` offset fields, nearest-neighbour and edge-clamped) вЂ” each `field`в†’`field`, composing freely before a terminal `FieldToFrame`. **Phase 3** adds coordinate-space transforms `FieldRotate` (rotate a field around its centre by an `angle` input + `spin` deg/sec, edge-wrapping) and `FieldTile` (repeat a field `tilesX`Г—`tilesY`); these are whole-field coordinate remaps (distinct from `FieldWarp`'s per-pixel offsets), so they ship as standalone composite nodes.

**FastLED helper nodes.** A handful of nodes wrap a single FastLED primitive so preview and firmware match by construction (evaluator + codegen dispatch, no shims):

- **`HeatColor`** (color) вЂ” `heat` (0вЂ“1) в†’ the black-body fire ramp. The evaluator reuses the shared `heatColor()` helper (also backing Fire2012), scaling 0вЂ“1 в†’ 0вЂ“255; codegen emits `HeatColor()`. Pair with `FieldToFrame`'s `heat` palette for a field-driven version.
- **`Rainbow`** (pattern) вЂ” FastLED `fill_rainbow`: a scrolling hue sweep, `deltaHue` sets the per-LED spread. `speed` is a normalised 0вЂ“1 slider (`SPEED_MAX.Rainbow = 120` hue-units/sec in `speedRange.ts`); index order matches the `[y*W+x]` buffer layout so preview lines up with the sketch.
- **`Gamma`** (composite) вЂ” perceptual gamma correction, `frame`в†’`frame`. Evaluator applies `255В·(c/255)^Оі` per channel; codegen emits `napplyGamma_video(buf, NUM_LEDS, Оі)`.
- **`Interval`** (math) вЂ” a metronome that emits a `bool` pulse every `interval` seconds (the non-audio analogue of `EVERY_N_MILLISECONDS`). Stateful in the evaluator (module-level `intervalLast`, keyed per group instance like the other stateful nodes); codegen emits a `static uint32_t` millis timer.

**Signal utility nodes.** Small float/trigger plumbing that makes signals composable: `Smooth` (low-pass EMA toward the input with a `response` time constant вЂ” the tool for taming jittery FFT bands or a PotInput, since the cycle guard forbids self-feeding a `Lerp`), `SampleHold` (latches `value` on each rising edge of `trigger` вЂ” `Random в†’ SampleHold в†ђ BeatDetect` is the "new random value every beat" idiom; `Random` alone re-rolls every frame), `Envelope` (rising trigger edge в†’ 1, linear decay to 0 over `decay` seconds вЂ” the generic float analogue of BeatFlash; wire through `Ease` for a curve), `Switch` (A/B float selector by a `sel` bool вЂ” unlike `Gate`, both branches are live signals), and `FrameSwitch` (the frame-typed A/B counterpart, composite category; falls back to the wired side when the selected one is empty, and evaluates both inputs so hidden stateful patterns keep advancing). The stateful three keep per-instance state in module-level maps (`smoothState`/`holdState`/`envState`, pruned like the rest) and their firmware mirrors use `millis()`-based statics seeded from the first sample, so preview matches on-device; all are dwell-time based, not framerate based.

**Trails (feedback buffer).** `Trails` (composite) is the canonical FastLED `fadeToBlackBy()`-and-accumulate idiom as a general-purpose node: it holds a persistent per-instance frame buffer (`trailState`, keyed like the other stateful nodes) that fades by `decay` each tick and is re-lightened per-channel wherever the incoming frame is brighter (`max(faded, input)`), so a moving highlight (a `Circle` driven by `Sin`/`Cos`, a `Blobs` charge, scrolling `Text`) leaves a comet trail instead of needing a pattern with trails baked in. Codegen mirrors the `Code` node's persistence trick: the node's own `buf_` render buffer is deliberately *not* reseeded from the input each frame (no `seedFrom`/`memmove`) вЂ” it fades and re-lightens in place via `fadeToBlackBy()` plus a per-channel `max`, exactly like the evaluator. Left untouched (not reset) while unwired so it resumes cleanly if reconnected; a graph with nothing ever wired in just renders solid black.

**Color Trails (stateful advection).** `ColorTrails` (pattern, Audio-Reactive) is adapted from Pygame prototypes by Stefan Petrick, creator of [AnimARTrix](https://github.com/StefanPetrick/animartrix); preserve that credit in UI descriptions, source, README/notices, and generated firmware. `injectionMode` can paint a palette-swept Lissajous segment, a continuously refreshed rainbow perimeter (the "liquid frame"), or both into the prior frame. `flowMode` selects the original scrolling CВІ 1D Perlin profiles or true 2D Perlin sampling with time on an independent axis, which makes the row/column displacement fields morph organically instead of merely translating. The Y profile shifts each row horizontally into an intermediate CRGB frame, the independently seeded/reversed X profile shifts each column vertically back into the persistent output, then `persistence` applies the very slow fade. Preview and codegen share the same integer gradient hashes, smootherstep, 3Г— line oversampling, soft endpoint discs, toroidal sampling, and elapsed-time scaling (the source is calibrated at 60 fps). Each horizontal and vertical displacement is capped to В±1 pixel per rendered frame, including audio boosts and delayed-frame scaling, preserving continuous subpixel blends instead of jumping across the trail. The X/Y controls retain the prototype defaults; `displacement` controls warp strength, `endpointSpeed` the line motion, and `colorSpeed` palette travel. Audio inputs are optional: bass strengthens displacement, mids accelerates endpoints, treble accelerates palette travel, and beat drives a short line-endpoint bloom, so autonomous motion remains intact unwired. Firmware declares `buf_cttmp_<id>` as an ordinary render buffer beside the node's persistent output, allowing MatrixOutput PSRAM to offload both; `estimateFirmwareRam` counts both CRGB frames. The initial catalog is Water, Polar Waves, RGB Blobs, Spiralus, and Complex Kaleido. `preview.ts` and `codegen.ts` implement paired normalized-coordinate renderers; keep their equations and smoothing constants aligned. Audio is optional, and unwired nodes remain autonomous. `audioAmount` sets overall reactivity depth: bass changes radial pressure/scale, mids changes angular flow, treble adds spatial detail and colour separation, kick/beat inject radial shocks, snare changes symmetry/arm counts, and hi-hat adds fine shimmer. Generated firmware must retain Stefan's name, upstream URL, and `CC BY-NC-SA 4.0` marker. Do not move derivative mathematics into core files or treat this module as MIT; if Stefan asks for removal, remove the node registration/imports and this directory without changing any microphone or SD-card analysis path.

**Frame Feedback / Delay.** `FrameFeedback` (composite) is the bounded recursive video-synth counterpart to `Trails`: it stores its own output in a fixed ring buffer, reads the output from `delayFrames` ticks ago, optionally fades and transforms that delayed frame (`none`/`translate`/`rotate`/`scale`), then blends it over the live input (`normal`/`screen`/`add`/`multiply`/`difference`/`lighten`). Because the recursion is internal to the node, the graph stays acyclic while still supporting feedback echoes, smeared rotations, and delayed trails. Preview state lives in `frameFeedbackState`; firmware emits a static `_fb_<id>[delayFrames + 1][NUM_LEDS]` ring and copies the produced `buf_<id>` back into it each loop. RAM estimates count that history as internal state: `(delayFrames + 1) Г— NUM_LEDS Г— sizeof(CRGB)`. It intentionally remains internal even when MatrixOutput's `usePsram` moves ordinary render buffers to PSRAM, matching the current stateful-node convention.

**Zones (named rectangular regions).** `Zones` (composite) is for installations with multiple logical display areas: it exposes four named zones (AвЂ“D, each with an `Enabled` toggle and a normalized 0вЂ“1 `X`/`Y`/`W`/`H` rectangle, grouped into collapsible sections via `PROPERTY_GROUPS`) and an optional `base` frame. It seeds its output from `base` (or black), then for each enabled zone with a wired frame, copies that frame's own pixels into the output only within the zone's rectangle вЂ” later zones (Aв†’D) paint over earlier ones where rectangles overlap, and a disabled or unwired zone is skipped entirely so partial wiring never punches a black hole in the output. Evaluator and codegen share the same seed-then-paint compositing (the C++ side loops over `WIDTH`/`HEIGHT`-scaled bounds baked from each zone's rectangle at codegen time).

**Fire / Fire2012 shared controls (direction, turbulence, paletteMix, mirror, seed).** Both fire nodes keep their own heat algorithm (`Fire`'s single-neighbor blur vs. `Fire2012`'s classic two-row-lookahead kernel) but share this exact set of five extra controls. Internally, the heat simulation always runs in a canonical primary/secondary grid вЂ” primary (`P`) is the distance from the flame base where sparks land, secondary (`S`) is the position across the flame's width вЂ” so `direction` (up/down/left/right) only has to remap `(p, s)` back to real `(x, y)` once, at the final palette-sampling step (`firePrimaryLen`/`fireSecondaryLen`/`fireToXY` in `graphEvaluator.ts`); codegen mirrors this with `fireGrid`/`fireXYExpr`, emitting `P`/`S` as the `WIDTH`/`HEIGHT` **macro names** rather than baked JS numbers so the heat array's size stays correct under MatrixOutput's supersample toggle. `turbulence` widens the sideways diffusion window (1 reproduces each node's original fixed kernel width exactly); `paletteMix` blends the palette colour toward plain heat-brightness grayscale; `mirror` folds the rendered frame symmetric across its width (columns for up/down, rows for left/right) as a post-process, same as the zones/mirror pattern elsewhere. `seed` (0 = free-running `Math.random`/`random8()`, unchanged from before this control existed) switches cooling/sparking to a deterministic per-instance LCG (`fireRandom` in the evaluator; a local `_fireLcg_<id>` static in codegen, since FastLED's global `random8()` can't be seeded per-node) so the same seed reproduces the same flame from a fresh start.

**Pride2015 / Pacifica / TwinkleFox (evocative Kriegsman homages).** `Pride2015`, `Pacifica`, and `TwinkleFox` (pattern, Generative) are named after the classic FastLED demos but вЂ” like `Plasma` above вЂ” are original evocative formulas rather than literal ports of the originals' 16-bit fixed-point/palette-blend/PRNG16 pipelines: `Pride2015` sweeps a full-spectrum rainbow hue across the matrix with a breathing sine brightness wave; `Pacifica` sums a few scrolling sine wave layers through a connected palette (default `ocean`) with a whitecap sparkle where a faster secondary wave crests; `TwinkleFox` gives each pixel an independent twinkle cycle via a deterministic per-index hash (`twinkleHash` in `graphEvaluator.ts`, mirrored verbatim in codegen), coloured from a connected palette (default `party`), with a `density` slider that softens the brightness curve from sparse sharp sparkles to most-pixels-lit. All three use the identical formula on the preview and firmware side (Plasma's approach, not Fire2012's "JS approximates, C++ is exact" split), so there's no algorithm mismatch to track вЂ” only the hue-to-color mapping differs cosmetically (`hsv()`/`samplePalette()` vs. `CHSV`/`ColorFromPalette`, same as Plasma). TwinkleFox is not yet hardware-validated.

**Saturation / RGBToHSV.** `Saturation` (composite) scales a frame's per-pixel HSV saturation by `amount` (0вЂ“2; 1 = unchanged, 0 = greyscale) вЂ” same inline RGBв†”HSV extraction as `HueShift` in the evaluator, `rgb2hsv_approximate()`/`CHSV` in codegen. `RGBToHSV` (color) is the inverse of `HSVToRGB`: given a connected color, it outputs `h`/`s`/`v` floats (e.g. to read the hue out of a sampled palette color).

**EncoderInput.** A rotary encoder (e.g. KY-040) alongside `ButtonInput`/`PotInput` вЂ” `pinA`/`pinB` quadrature pins plus `pinSW` for the integrated push-button. The preview is an inert stub (`position: 0, pressed: false`, same convention as the other two hardware-input nodes вЂ” there's no real hardware to read); firmware does a polling quadrature decode (a standard 4-bit lookup table, no interrupts, matching this codebase's existing interrupt-free hardware-input style) into an unbounded running `position` count вЂ” wire it through `MapRange`/`Mod` to normalise or wrap.

**Field Noise / Frame в†’ Field.** `FieldNoise` (field) is an organic fBm noise source straight as a scalar field вЂ” the same octave-summed construction as the `FractalNoise` pattern node (JS: `_snoise2`; firmware: `inoise8`, matching that node's existing preview/firmware noise-algorithm mismatch), just written to the field buffer instead of sampled through a palette, so it sits beside `FieldFormula` as a second raw field generator. `FrameToField` (field) is the inverse of `FieldToFrame`: it extracts a 0вЂ“1 brightness field from a rendered frame (average of r,g,b, the same convention `Mask` uses for a mask frame's opacity), so any pattern's output вЂ” not just formulas вЂ” can drive a downstream `FieldWarp`/`FieldMath`/mask.

**MatrixOutput power cap.** MatrixOutput gained an optional `powerLimit` toggle plus `volts`/`milliamps` (the volts/mA editors are gated on the toggle via `isPropertyEnabled`). When on, `cppGenerator.ts` emits `FastLED.setMaxPowerInVoltsAndMilliamps(volts, milliamps)` in `setup()` so FastLED auto-dims to keep PSU draw under the cap. Preview-only (no effect on the live render).

**MatrixOutput hardware settings (FastLED parity).** MatrixOutput also carries `brightness` (0вЂ“255, default 200 вЂ” the previously hardcoded `FastLED.setBrightness` value; a `PROPERTY_META_OVERRIDES` entry since the shared `brightness` meta is 0вЂ“1), `correction` (`none`/`TypicalLEDStrip`/`TypicalPixelString` в†’ `FastLED.setCorrection`), `dither` (default on = FastLED's default, emits nothing; off emits `setDither(DISABLE_DITHER)`), `overclock` (1вЂ“1.7Г— в†’ `#define FASTLED_OVERCLOCK` **before** the FastLED include; clockless chipsets only), and `clockPin` (SPI chipsets only в†’ `CLOCK_PIN` in `addLeds`). The chipset dropdown spans clockless (WS2812B/WS2811/WS2815/SK6812/**SK6812-RGBW**/WS2816/SM16824E/NEOPIXEL) and SPI (APA102/**APA102HD**/WS2801/HD108) parts; `SK6812-RGBW` maps to `SK6812` + `.setRgbw(RgbwDefault())`, and `NEOPIXEL` omits the colour-order template arg (its FastLED alias hardcodes GRB). One shared helper family in `cppGenerator.ts` вЂ” `ledHardwareFromProps` (sanitises against the `CHIPSET_OPTIONS`/`COLOR_ORDER_OPTIONS`/`CORRECTION_OPTIONS`/`SPI_CHIPSETS` lists exported from `nodeLibrary.ts`, since these strings land in C++ template args) + `fastledSetupCpp` + `overclockDefineCpp` вЂ” feeds all three sketch generators (normal, show controller, music-sync player; the player skips the brightness line because show events drive it). `isPropertyEnabled` gates `clockPin` to SPI chipsets and `overclock` to clockless ones. **Preview:** the master brightness is mirrored on the terminal frame (`applyMasterBrightness` in `LEDPreview.tsx`) so preview matches firmware; correction/dither are deliberately *not* previewed (they compensate the physical LEDs so hardware matches the intent the preview already shows). Not yet hardware-validated.

**MatrixOutput physical layout (`layout`/tiling/custom XY map).** MatrixOutput's `layout` property (`matrix` default / `strip` / `panels` / `custom`) generalises the old serpentine-only wiring remap into `src/state/xyLayout.ts`'s `buildXYTable(width, height, props)`, a pure function returning a gridв†’physical-LED-index lookup table (or `null` when a plain row-major memmove already does the right thing). `panels` splits the WIDTHГ—HEIGHT grid into `tilesX`Г—`tilesY` equal panels; each panel can be rotated 0/90/180/270В° via the comma-separated `tileRotations` (row-major panel-grid order) and the panel chain itself can snake via `tileSerpentine` вЂ” the existing pixel-level `serpentine` flag still governs the zig-zag *within* each panel (or the whole grid when `tilesX`/`tilesY` are both 1). `custom` takes an explicit JSON permutation of `0..N-1` in `customXYMap`. Invalid layout input is now surfaced explicitly by `validateMatrixLayout()` / `findMatrixLayoutErrors()` (panel divisibility, bad rotation entries/count, bad custom-map JSON/permutations), and `MatrixOutputUpload` blocks upload/export actions until it's fixed; `buildXYTable()` still keeps the old row-major fallback as a last-resort safety net so codegen never emits a broken table. `isPropertyEnabled` in `nodeLibrary.ts` gates the tile/custom-map fields to their relevant `layout` value. Codegen (`cppGenerator.ts`) bakes the table once as a `const uint16_t _xytable[] PROGMEM` plus an `XY(x,y)` reader вЂ” a single code path that replaced the old serpentine-only formula, so pixel serpentine, panel tiling, and a custom map compose freely. Since physical wiring order never changes *what* renders at a given (x,y) вЂ” only which LED index it's written to вЂ” the live preview needs no changes to stay correct for the pixel content itself. `LEDPreview.tsx` does draw a purely cosmetic panel-boundary overlay for valid `layout === 'panels'` setups: `matrixTileLayout()` (`graphStore.ts`, memoised like `matrixDims`) returns the `tilesX`Г—`tilesY` grid only when the panel layout validates, and a second `<canvas>` stacked on top of the main one (`.canvasStack`/`.tileGridOverlay` in `LEDPreview.module.css`) is redrawn only when the tile grid or canvas size changes вЂ” not every animation frame вЂ” so it costs nothing during normal playback; tile rotation/chain order isn't reflected since neither affects the rendered content. Not yet hardware-validated.

**MatrixOutput PSRAM buffers (`usePsram`/`psramMode`).** For designs whose static buffers overflow the ESP32's internal DRAM (`dram0_0_seg` link errors), MatrixOutput carries a **Use PSRAM** toggle rendered in `MatrixOutputUpload` вЂ” *not* the generic property list, because visibility depends on the selected board: `Board.psram` in `uploadStore.ts` lists each board's PSRAM FQBN options (S3: OPI `PSRAM=opi` / QSPI `PSRAM=enabled`, picked via `psramMode` since the interface is a module-package fact that can't be probed from the host; plain ESP32: QSPI only). When on, both generators declare the per-node `buf_`/`field_` render buffers (and the show sketch's `showA`/`showB` + per-pattern buffers) as pointers allocated via the shared `_psAlloc` helper (`PSRAM_ALLOC_CPP`/`psramBufferDecl` in `cppGenerator.ts`) вЂ” `ps_malloc` with internal-heap fallback via a runtime `psramFound()` check вЂ” and `runUpload` appends the chosen PSRAM option to the FQBN. `leds` always stays a static internal array (FastLED's ESP32 drivers read it from ISR/DMA context where PSRAM access can fault). The `psramAllowed: false` codegen opt (passed when the selected board has no PSRAM) neutralises a stale toggle so AVR/RP2040 sketches never see ESP32-only calls. Stateful nodes' `static` in-loop arrays (ReactionDiffusion, GameOfLife, вЂ¦) still live in `.bss` вЂ” a follow-up. Not yet hardware-validated. Related: the upload helper's failure banner is phase-aware (`_compile_upload` returns `(rc, phase)`) вЂ” a build failure says nothing reached the board; only an upload failure suggests download mode.

**Firmware RAM estimate.** `estimateFirmwareRam()` (`src/utils/validateGraph.ts`) gives a pre-upload RAM budget: it walks backward from MatrixOutput through `edges` to find every node that actually feeds the sketch (an unreached node gets no buffer in codegen, so it's excluded), sums the physical `leds` array plus each reachable `frame`/`field` render buffer (including ColorTrails' second CRGB advection buffer), and adds known per-LED overhead for the handful of simulation nodes whose `static` state is bigger than their own render buffer (Fire2012's heat map, Game of Life's cell/next/bright grids, Reaction-Diffusion's four u/v grids, WaveSim's p/c/n grids) plus Particles' fixed-size pool (independent of matrix size). Since that stateful state always stays in internal `.bss` regardless of PSRAM (see above), the estimate is split into `internalBytes`/`psramBytes` by MatrixOutput's `usePsram` toggle. `MatrixOutputUpload` shows a compact KB readout next to the power estimate, and `validateGraph` warns when the internal estimate crosses a conservative heads-up threshold, nudging toward enabling PSRAM. It's a rough guide, not a linker-accurate figure вЂ” the post-compile flash/RAM report from the build engine remains the ground truth, which the live controller-capacity meter below surfaces directly rather than estimating.

**Live controller-capacity meter.** `POST /api/compile-check` (`backend/app.py`) runs a real compile-only build вЂ” same engine, same sketch generator, no port so nothing is flashed вЂ” via the existing `_compile_upload`/`_compile_upload_fbuild` generators (already support `port=""` as "compiled only"), drained synchronously by `_drain_compile` into one JSON result instead of a streamed log: `_size_bytes_report` (arduino-cli: `Sketch usesвЂ¦`/`Global variables useвЂ¦`) and `_fbuild_size_bytes_report` (fbuild: `Flash:`/`RAM:`, unit-converting KB/MB/bytes) return `{usedBytes, limitBytes, percent}` per flash/RAM, and `overflow` reuses the existing linker-overflow marker check. Two fbuild-specific correctness fixes, both found by live-testing against the real toolchain: (1) fbuild's project scaffold is one shared directory, so two overlapping compiles (a live re-check racing another check or a real Upload) used to interleave `main.cpp` writes and build output and corrupt each other's result вЂ” `_fbuild_build_lock` now holds for the whole duration of every `_compile_upload_fbuild` run, serializing all fbuild compiles project-wide. (2) fbuild only reprints its `Flash:`/`RAM:` line when it actually reruns the linker, so a no-op incremental build (nothing changed since the last check вЂ” the common case once the debounce settles) silently produced no size report at all; `_fbuild_cached_size()` falls back to fbuild's own persisted `.fbuild/build/<env>/release/.firmware_size_cache.json` when the live build output has none. `src/state/capacityStore.ts` debounces requests 1200 ms and caches by board target (fqbn + any PSRAM option) plus a hash of the generated sketch вЂ” switching boards invalidates immediately so a reading is never shown against the wrong target вЂ” exposing `checking`/`measured`/`stale`/`toolchain-missing`; a completed 'measured' result's own `ok`/`overflow` flags (read via `src/utils/capacityFormat.ts`'s `summarizeCapacity`) distinguish a healthy percentage from "won't fit" from an unrelated compile error, so there's no separate status enum value per outcome вЂ” that same tooltip surfaces the compile-check's log tail on failure for debugging. On overflow, `summarizeCapacity` shows the actual over-100% usage in red (`SRAM 112%`) rather than a bare "won't fit" when a figure is available. A genuine overflow is normally a **hard linker failure** вЂ” `ld` refuses to produce an `.elf` at all, so fbuild never reaches the step that prints its own `Flash:`/`RAM:` summary (confirmed with a real forced ESP32-S3 DRAM overflow: no size line anywhere in the log, only `ld`'s `region 'dram0_0_seg' overflowed by N bytes`). `_fbuild_overflow_estimate()` derives a real percentage from that same linker message plus fbuild's own board-memory line (`Memory: 8.00MB Flash, 320.00KB RAM`, printed up front regardless of outcome) вЂ” region names containing "ram" attribute to RAM, everything else to flash вЂ” wired in as `/api/compile-check`'s last fallback when a failed compile produced no size line at all. Falls back to the plain "won't fit" text only when even the linker gave no parseable overflow amount. `summarizeCapacity` always reports **both** metrics together (`flash P%`/`flash n/a` and `SRAM P%`/`SRAM n/a`) in every outcome rather than showing whichever one happened to be available вЂ” a design flipping from "SRAM 101%" (failing to link) to just "flash 10%" (succeeding) used to silently swap which metric the meter displayed (successful ESP32/fbuild builds routinely self-report an "impossible" RAM figure that `_fbuild_cached_size` discards as unreliable, while a failed build's overflow percentage has no such guard), reading as "the RAM problem is fixed" when RAM had simply stopped being measurable; pairing both figures unconditionally means that gap can never be mistaken for good news. `MatrixOutputUpload.tsx` (the always-mounted node body, not just the upload popup) generates the sketch and drives the check continuously while composing, rendering the compact meter (`Arduino Uno В· flash 74% В· SRAM 41%`); `MatrixOutputDeployPopup.tsx` reads the same store passively (repeat requests with an unchanged board+code are no-ops in the store, so both components triggering costs nothing extra) and folds a confirmed overflow into its `blockingErrors`, making the measured result вЂ” not the fast heuristic above вЂ” the authority for whether Upload is allowed. `PatternCollectionBody.tsx` shows the flash/RAM delta between the last two same-board measurements (`capacityDelta`/`formatCapacityDelta`), so adding or removing a pattern surfaces what it cost once the next check lands. Not yet hardware-validated.
